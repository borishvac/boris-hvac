# Чек-лист для разработчика

Пошаговая проверка перед запуском калькулятора на сайте.

## Этап 1: Бэк-парсер (GitHub Actions)

- [ ] **Создан репозиторий** на GitHub (приватный или публичный — без разницы), содержимое архива загружено в корень.
- [ ] **Settings → Pages**: источник = `Deploy from a branch`, ветка = `main`, папка = `/(root)`. Подождать 1–2 мин, открыть `https://USER.github.io/REPO/catalog.json` — должен отдаваться JSON.
- [ ] **Settings → Actions → General → Workflow permissions**: выбрать `Read and write permissions`, сохранить.
- [ ] **Actions → Update Breezart catalog → Run workflow**. После завершения проверить:
  - [ ] В репозитории появился свежий коммит от `github-actions[bot]`.
  - [ ] В `frontend/catalog.json` версия (поле `version`) обновилась на сегодняшнюю дату.
  - [ ] В логе шага «Run parser» написано что-то вроде `OK: найдены серии: lux(N), lux_f(N), ..., aqua(N), ...` и `OK: записан ...catalog.json (N моделей суммарно)`.
- [ ] **Запланированный запуск**: workflow стоит на cron `0 3 1 * *` (1-е число каждого месяца, 03:00 UTC). Можно поменять, отредактировав `.github/workflows/update-catalog.yml`.

**Что делать, если парсер падает.** Скорее всего, Breezart поменяла структуру PDF (заголовок серии, формат строки прайса). В лог Actions будет понятная ошибка. Править регулярки и список `SERIES_MARKERS` в `backend/parser.py`. Пока парсер сломан, `catalog.json` остаётся прежним (последний удачный) — клиенты не пострадают.

## Этап 2: Tilda webhook (приём лидов)

Калькулятор после заполнения формы шлёт POST с полями:
```
Name, Phone, Form, Package, Total, Area, Rooms, Property, Location
```

Варианты, куда подключить:

**Вариант А — встроенная форма Tilda Webhook.**
1. В Tilda создать обычную форму (любой блок T754 / T755).
2. В её настройках указать «Сервис приёма данных» = вашу почту, CRM (Bitrix24, amoCRM) или Telegram.
3. Через DevTools на странице с формой отправить тестовую заявку, в Network найти POST на `forms.tildacdn.com/procces/` — это и есть нужный URL.

**Вариант Б — внешний вебхук (Make / Zapier / своя интеграция).**
1. Создать вебхук в Make/Zapier.
2. Указать его URL в `TILDA_WEBHOOK_URL`.
3. Из вебхука настроить маршрутизацию в нужную CRM.

- [ ] URL вебхука получен.
- [ ] Сделан 1 тестовый запрос (через DevTools или Postman) — лид дошёл до CRM/почты.

## Этап 3: Настройка `calculator.html`

Открыть файл и в начале блока `<script>` (примерно строки 296–320) исправить:

```js
const CATALOG_URL       = "https://USER.github.io/REPO/catalog.json";   // из этапа 1
const TILDA_WEBHOOK_URL = "https://forms.tildacdn.com/procces/";        // из этапа 2

const COMPANY = {
  name:  "ООО «Ваша компания»",
  phone: "+7 (495) 123-45-67",
  email: "info@yourcompany.ru",
  site:  "yourcompany.ru",
};
```

- [ ] `CATALOG_URL` указывает на ваш GitHub Pages.
- [ ] `TILDA_WEBHOOK_URL` указывает на ваш вебхук.
- [ ] Реквизиты компании заполнены.

## Этап 4: Вставка в Tilda

1. На нужной странице добавить блок **«Другое» → T123 «HTML-код»**.
2. Скопировать всё содержимое `calculator.html` целиком (от первой строки `<!--` до последней `</script>`).
3. Вставить в блок T123.
4. Опубликовать страницу.

- [ ] Блок T123 добавлен.
- [ ] Калькулятор отображается, без ошибок в DevTools → Console.
- [ ] Под калькулятором (внизу справа) видна строка «Цены Breezart актуальны на ...».

## Этап 5: Приёмочное тестирование

Прокликать минимум 3 сценария и убедиться, что цифры разумны:

### Сценарий А — типовой
- Дом, 200 м², 4 комнаты, 2 тех., 4 человека, газ, Москва, без бассейна/животных/аллергий.
- **Ожидание:** Базово ~5.5М ₽, Комфорт ~6М ₽, Люкс ~8М ₽. Подобрана **Breezart 550 Aqua** в Базово/Комфорт и **1000 Aqua F** в Люкс.

### Сценарий Б — небольшая квартира
- Квартира, 100 м², 3 комнаты, 2 тех., 3 человека, Москва.
- **Ожидание:** ~3М / 3.4М / 4.3М ₽. Приточка **Lux** (электр.).

### Сценарий В — большой дом со спецтребованиями
- Дом, 500 м², 8 комнат, 4 тех., 6 человек, газ, МО, животные ✓, аллергия ✓.
- **Ожидание:** ~10М / 11М / 15М ₽. В каждом пакете строка «Секция фильтра E11 (HEPA)». В Люксе строка «Озонация» +300 000 ₽. Транспорт ×2 (выше, чем в Москве).

### Скачивание PDF
- [ ] Нажать «Скачать смету» на любой карточке.
- [ ] Появилась форма «Имя + Телефон».
- [ ] После заполнения PDF скачивается, открывается, **кириллица отображается корректно**.
- [ ] В CRM/на почте появился лид с правильными данными.

### Если кириллица в PDF в виде квадратиков

Значит шрифт PT Sans не подгрузился с CDN (`cdn.jsdelivr.net` или `unpkg.com`). Лечится за 10 минут:

1. Скачать два TTF-файла:
   - `https://cdn.jsdelivr.net/fontsource/fonts/pt-sans@latest/cyrillic-400-normal.ttf` → переименовать в `pt-sans-regular.ttf`
   - `https://cdn.jsdelivr.net/fontsource/fonts/pt-sans@latest/cyrillic-700-normal.ttf` → переименовать в `pt-sans-bold.ttf`
2. Положить оба файла в папку `frontend/` репозитория (рядом с `catalog.json`).
3. Закоммитить и запушить. После деплоя на GitHub Pages они будут доступны по URL вида `https://USER.github.io/REPO/pt-sans-regular.ttf`.
4. В `calculator.html` найти массивы `FONT_URLS_REGULAR` и `FONT_URLS_BOLD`, **первой строкой** в каждый добавить URL вашего файла:
   ```js
   const FONT_URLS_REGULAR = [
     "https://USER.github.io/REPO/pt-sans-regular.ttf",  // <— добавить
     "https://cdn.jsdelivr.net/fontsource/fonts/pt-sans@latest/cyrillic-400-normal.ttf",
     "https://unpkg.com/@fontsource/pt-sans/files/pt-sans-cyrillic-400-normal.ttf",
   ];
   ```
5. Обновить блок T123 в Tilda и переопубликовать страницу.

## Этап 6: Эксплуатация

- **Раз в месяц** автоматически обновляется `catalog.json` — ничего делать не нужно.
- **При повышении цен Breezart** парсер их подхватит самостоятельно при следующем запуске.
- **Чтобы поменять свои коммерческие правила** (наценка, тарифы, лимиты) — править объект `RULES` в начале `<script>` в `calculator.html` и переопубликовать блок T123.
- **Мониторинг падений парсера:** GitHub автоматически шлёт email владельцу репозитория при падении workflow. Можно настроить нотификации в Slack/Telegram через [GitHub Actions notification integrations](https://docs.github.com/en/actions/managing-workflow-runs-and-deployments/managing-workflow-runs/notifications-for-workflow-runs).

## Известные ограничения

- **Объекты с бассейном** — выводится дисклеймер «считается отдельно», расчёт даёт смету без бассейна. Если эта категория важна, нужна отдельная логика подбора серий Pool / Aqua Pool DH.
- **ПВУ с рекуператорами** (Lux RP, Lux RR, Aqua RP, Aqua RR) — не входят в подбор. Если потребуется, дорабатывать `pickIntakeUnit` в `calculator.html`.
- **Аэродинамика** — точные кривые «расход vs давление» от Breezart не используются (их БД не в открытом доступе). На практике для типовых сетей это не критично — приточки имеют запас.
- **Цены в Москве/МО** — единственная разница в формуле — транспорт ×2. Никаких других региональных надбавок (на ПНР, монтаж и т.п.) нет.
